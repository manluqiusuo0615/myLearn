# 算法分析
主要有两个方面：   
    正确性：算法功能与问题的要求一致？
            数学证明？可不那么简单。。。  
      成本：运行时间 + 所需存储空间（时间成本和空间成本）
            如何度量？如何比较     
## 建模：   
Ta（P） = 算法A求解问题实例P的计算成本       
同样一个问题，可能出现的问题实例太多，如何归纳概括？
数学给我们提供了常用的，非常有效的方法：划分等价类。（需要对任何一个问题无数的实例，进行一定的粗分类,并且就某一类谈它的计算成本）     
观察：问题实例的规模，往往是决定计算成本的主要因素。      
通常：规模接近，计算成本也接近。随着规模的增加，计算成本也上升   
## 特定算法  + 不同实例
把原来每一个具体的问题实例P变成了一个笼统而言的一个规模的度量值n   
令Ta(n) = 用算法a求解某一问题规模为n的实例，所需要的计算成本，讨论特定算法a(及其对应的问题)时，简记作T(n)   
对于同一问题等规模的不同实例，计算成本也不尽相同，甚至有实质差别。   
既然如此，又该如何定义T(n)呢？   
稳妥起见，取 T(n) = max { T(P) |  |P| = n}   
在所有规模为n的问题实例中，将它们的计算成功进行总体的比较，并取出其中的最大值   
亦即，在规模同为n的所有示例中，只关注最坏(成本最高)着   
## 理想模型  特定问题 + 不同算法
同一个问题通常有多种算法，如何评判其优劣？
实验统计是最直接的方法，但足以准确反映算法的真正效率？
不足够：
- 不同的算法，可能更适应于不同规模的输入
- 不同的算法，可能更适应于不同类型的输入
- 同一算法，可能由不同程序员，用不同程序语言，经不同编译器实现
- 同一算法，可能实现并运行在不同的体系结构，操作系统。。。。。。    
为给出客观的评判，需要抽象出一个理想的平台或模型   
不在依赖于上述种种具体的因素    
从而直接而准确地描述，测试并评价算法
## 图灵机模型
Tape(带)：依次均匀地划分为单元格(cell)，各注有某一字符，默认为'#'    
Alphabet字符的种类有限    
head: 总是对准某一单元格，并可读取和改写其中的字符，每经过一个节拍，可转向左侧或右侧的邻格   
State TM:总是处于有限种状态种的某一种，每经过一个节拍，可(按照规则)转向另一种状态（这个状态本身只有有限种可能）   
Transition Function（规则）:(q,c;d,L/R,p)     
q代表当前读写头所处的状态，c代表的是当前读写头所正对的单元里所存的字符，   
前两项可以理解为是当前的状态，后面三项描写的是头紧接着的动作，也就是我们说的转换   
d代表的是在当前单元格里填入的或者说是修改成的新字符
L/R代表修改后可以向左或右移动一个单元格
同时将自己的状态由开始的q转换为p
若当前状态为q且当前字符为c，则将当前字符改写成d,转向左侧/右侧的邻格;转入P状态一旦转入特定状态'h',则停机  
TM：Increase  将二进制非负整数加一   
(<,1,0,L,<)//左行，1->0
(<,0,1,R,>)//掉头,0->1
(<,#,1,R,>)//?
(<,0,0,R,>)//右行
(<,#,#,L,h)//复位
## RAM模型 Random Access Machine
寄存器顺序编号，总数没有限制  
R[0],R[1],R[2],......    
每一基本操作仅需常数时间     
R[i] <- c             R[i] <- R[R[j]]       R[i] <- R[j]+R[k]    
R[i] <- R[j]          R[R[i]] <- R[j]       R[i] <- R[j]-R[k]    
GOTO 1      STOP    
判断是不是0   IF R[i] = 0 GOTO 1    
判断是不是正数   IF R[i]  > 0 GOTO 1    
与TM模型一样，RAM模型也是一般计算工具的简化与抽象   
使我们可以独立于具体的平台，对算法的概率做出可信的比较与批判     
在这些模型种：
算法的运行时间   转换为了   算法需要执行的基本操作次数      
T(n) = 算法为求解规模为n的问题，所需要执行的基本操作次数    

执行过程可以记录为一张表   
表的行数即是所执行基本执行的总条数    

上面通过图灵机和RAM模型，为度量算法性能，提供了准确的尺度(执行的次数)
